# Chapter 5.2 — Cache Layers

> In the previous chapter, we understood **why caching exists**, how a cache works, cache hits, cache misses, TTL, stale data, cache consistency, cache stampede and request coordination.
>
> But one important question still remains:
>
> **Where do these caches actually exist?**
>
> Is there just one cache?
>
> Or are there multiple caches throughout the request journey?
>
> In reality, modern systems don't rely on a single cache.
>
> They cache data at multiple layers, each trying to stop the request as early as possible.

---

# Core Principle

Every cache layer has exactly one objective:

> **Can I satisfy this request so that the next layer doesn't have to do any work?**

If the answer is yes, the request stops there.

If not, it moves deeper into the system.

As we go deeper:

- Latency increases
- Network cost increases
- CPU usage increases
- More components wake up
- Database becomes more likely to be touched

So every cache layer acts as another opportunity to stop unnecessary work.

---

# Complete Request Journey

```text
Browser
    │
Browser Cache
    │
(Cache Miss)
    ▼
CDN / Edge Cache
    │
(Cache Miss)
    ▼
Reverse Proxy Cache
    │
(Cache Miss)
    ▼
Application Cache
    │
(Cache Miss)
    ▼
Distributed Cache
    │
(Cache Miss)
    ▼
Database (Source of Truth)
```

Every layer asks exactly the same question:

> "Can I answer this request?"

If yes → Stop.

If no → Forward.

# 1. Browser Cache

Suppose we visit:

```text
https://youtube.com
```

The browser downloads:

- HTML
- CSS
- JavaScript
- Images
- Fonts

Now imagine we refresh the page after a few seconds.

Should the browser download everything again?

Of course not.

It already has many of these resources.

So before sending any network request, the browser first checks its own cache.

If the resource exists and is still valid:

```text
Browser
    │
Browser Cache (Hit)
    │
Return Resource
```

The request never leaves the user's machine.

No network.

No bandwidth.

No server.

No CDN.

Nothing.

This is the cheapest cache in the entire system.

---

## Why Browser Cache Exists

Without browser cache:

Every refresh would download:

- CSS
- JS
- Images
- Fonts

again and again.

This wastes:

- bandwidth
- latency
- server resources

Browser cache eliminates all of that.

---

## Engineering Principle

> The cheapest request is the one that never leaves the browser.

This is the first cache layer.

---

# 2. OS & DNS Cache (Recap)

We already studied DNS in previous chapters.

Now observe it from a caching perspective.

Suppose the browser wants:

```text
youtube.com
```

It needs an IP address.

Should DNS resolution happen every single time?

No.

The browser maintains a DNS cache.

The operating system also maintains a DNS cache.

So before contacting external DNS servers, cached DNS entries are checked.

Although DNS caching is different from content caching, the underlying idea is identical.

Instead of caching HTML or images...

we cache:

```text
Domain
↓

IP Address
```

This avoids repeated DNS lookups.

---

# 3. CDN / Edge Cache

Now suppose the browser cache misses.

The request leaves the user's machine.

Imagine YouTube has edge servers in:

- Delhi
- Mumbai
- Singapore
- London
- New York

A user sitting in Gurgaon types:

```text
youtube.com
```

Question:

How does the browser know which server to connect to?

The browser only knows:

```text
youtube.com
```

It has no idea that these edge servers exist.

So who tells it?

The answer comes from something we already studied.

**DNS.**

DNS can return different IP addresses to different users.

A user in India may receive the IP of a nearby edge server.

A user in London may receive a completely different IP.

Thus, DNS naturally helps route users to geographically nearby cache servers.

---

## Why not directly connect to YouTube's data center?

Suppose YouTube has a single data center in California.

Every request from India would have to:

- Leave the browser
- Traverse the ISP
- Travel across the Internet
- Reach California

even for something like:

```text
logo.png
```

That is unnecessary.

Instead, place cache servers closer to users.

Now the request becomes:

```text
Browser
      │
      ▼
Nearby Edge Server
```

Much lower latency.

Much less bandwidth.

Far less load on YouTube's origin infrastructure.

---

# What should be cached here?

Initially, we might think:

Only static files.

Like:

- CSS
- JavaScript
- Images
- Fonts

These are excellent cache candidates because every user receives exactly the same response.

Examples:

```text
player.js
logo.png
styles.css
Roboto.ttf
```

One cached copy can serve millions of users.

This gives an incredible return on cache memory.

---

## Static vs Dynamic is NOT the correct mental model

Initially, it feels like:

Static → Cache

Dynamic → Don't cache

But that's not actually true.

The better mental model is:

> **Shared vs Personalized**

Example:

```http
GET /api/video/abc123
```

Response:

```json
{
    "title": "...",
    "duration": 900,
    "views": 1200000
}
```

This response is generated dynamically.

But every user receives almost the same response.

So it can still be cached.

Now consider:

```http
GET /me
```

Response:

```json
{
    "name": "Ujjwal",
    "history": [...]
}
```

This is personalized.

Another user must never receive it.

So shared caches should generally avoid caching such responses.

Thus the correct question is not:

> Is it static?

Instead ask:

> Can this response be safely shared among many users?

---

# HTML vs Assets

Consider the following HTML:

```html
<html>

<head>
    <link rel="stylesheet" href="/static/main.css">
    <script src="/static/player.js"></script>
</head>

<body>

<img src="/logo.png">

</body>

</html>
```

Notice something important.

The HTML itself does not contain:

- CSS
- JS
- Images

Instead it contains references.

Once the browser receives this HTML, it makes additional requests:

```text
GET /static/main.css

GET /static/player.js

GET /logo.png
```

These requests can be served directly by the CDN.

The browser doesn't care who serves them.

It only cares that:

- URL matches
- Response is correct
- Response is fast

---

# CDN caches HTTP Responses

One important realization:

A CDN is not fundamentally caching files.

It is caching **HTTP responses**.

Sometimes those responses happen to be:

- CSS
- JS
- Images
- Fonts
- Video chunks

Sometimes they are:

- Public JSON APIs
- Product metadata
- Public repository pages
- Trending content

As long as the response is safely reusable, it becomes a cache candidate.

---

# Cache Miss at CDN

Suppose the CDN doesn't have:

```text
thumbnail.jpg
```

Should it return:

```text
404
```

No.

Instead:

```text
Browser

↓

CDN

↓

Cache Miss

↓

Fetch from Origin

↓

Store Locally

↓

Return Response
```

The next request becomes a cache hit.

---

# Why not cache everything?

Consider:

```http
GET /watch?v=abc123
```

Suppose the generated HTML contains:

- Hello Ujjwal
- Recommendations
- Watch History
- Premium Badge

If the CDN caches this entire HTML...

The next user may receive:

```text
Hello Ujjwal
```

This is not just stale data.

This is a security issue.

The CDN incorrectly assumed:

> Every request for this URL has the same response.

That assumption was false.

---

## Important Principle

Shared caches work best when:

One cached copy

↓

Serves thousands or millions of users.

Examples:

- Images
- CSS
- JavaScript
- Fonts
- Video chunks
- Public metadata

Poor candidates:

- User dashboard
- Personal profile
- Order history
- Notifications

These are usually better handled deeper inside the system.

---

# CDN Summary

Purpose:

Stop requests before they ever reach the application's infrastructure.

Key ideas:

- Distributed globally
- DNS helps users reach nearby edge servers
- Cache shared HTTP responses
- Fetch from origin on cache miss
- Excellent for highly reusable content
- Think **Shared vs Personalized**, not **Static vs Dynamic**

---

# 4. Reverse Proxy Cache

Suppose the CDN misses.

Should every request immediately wake up the application?

No.

There is another opportunity to stop the request.

```text
Browser

↓

CDN (Miss)

↓

Reverse Proxy / Nginx

↓

Application
```

Since Nginx already sits in front of the application, it can maintain its own cache.

If Nginx has the response:

```text
Browser

↓

CDN (Miss)

↓

Nginx (Hit)

↓

Return Response
```

The application never executes.

CPU is saved.

Memory allocations are avoided.

Business logic never runs.

Application cache isn't checked.

Database is never touched.

Everything below Nginx stays asleep.

---

## Why cache here if CDN already exists?

Because the two caches are not competing.

They are cooperating.

Suppose only Nginx caches.

The next request still has to:

- Reach YouTube's infrastructure
- Travel to the data center
- Reach Nginx

Application is saved.

Network latency still exists.

Now suppose both cache.

The CDN serves the response.

Nginx never wakes up.

Application never wakes up.

Database never wakes up.

Always stop the request as early as possible.

This principle keeps repeating throughout the system.

## Multiple Cache Layers Work Together

One important realization:

These cache layers are **not competing**.

They are cooperating.

Think of the request flow as multiple checkpoints.

```text
Browser
    │
Browser Cache
    │
"Can I answer?"
    │
No
    ▼
CDN Cache
    │
"Can I answer?"
    │
No
    ▼
Reverse Proxy Cache
    │
"Can I answer?"
    │
No
    ▼
Application Cache
    │
"Can I answer?"
    │
No
    ▼
Distributed Cache
    │
"Can I answer?"
    │
No
    ▼
Database
```

Every layer has exactly one objective:

> **Stop the request as early as possible.**

---

# 5. Application Cache (In-Memory)

Suppose:

- Browser Cache misses.
- CDN misses.
- Reverse Proxy misses.

Now the request reaches the application.

Should the application directly query the database?

Again,

No.

Suppose the application frequently needs:

- User Profile
- Video Metadata
- Product Details
- Feature Flags
- Permissions

Fetching these repeatedly from the database is still expensive.

So the application keeps frequently accessed business objects in its own memory.

Example:

```text
Application

├── User Cache
├── Product Cache
├── Video Cache
├── Feature Flag Cache
└── Permission Cache
```

If the required object already exists:

```text
Application

↓

Memory Cache (Hit)

↓

Return Response
```

Database never executes.

---

## Why not let Nginx cache everything?

Because Nginx knows very little.

It only understands HTTP requests and responses.

The application understands business objects.

Examples:

- Users
- Orders
- Videos
- Permissions
- Inventory
- Recommendations

These are application-specific concepts.

Therefore, application cache is much more intelligent than a reverse proxy cache.

---

# 6. Distributed Cache (Motivation)

Application cache works well...

until we scale.

Imagine:

```text
Application Server A

Memory Cache
```

Now the Load Balancer sends the next request to:

```text
Application Server B
```

Its memory cache is completely different.

Cache Miss.

Suppose:

```
Server A

User Profile Cached
```

Next request:

```
Server B

No Cache
```

Same database query happens again.

Now imagine:

10

20

100

1000

Application servers.

Each one maintains its own memory.

The cache becomes fragmented.

This naturally creates a new problem.

We need one cache that every application server can access.

That idea motivates a **Distributed Cache**.

We are **not** discussing Redis yet.

At this stage, simply understand why a shared cache becomes necessary.

Redis will be introduced later as one implementation of a distributed cache.

---

# Cache Layers Summary

Each cache layer knows different things.

| Layer | Knows About | Best Used For |
|--------|-------------|---------------|
| Browser Cache | User's previously downloaded resources | CSS, JS, Images, Fonts |
| OS / DNS Cache | Domain → IP mapping | DNS lookups |
| CDN Cache | Shared HTTP responses | Static assets, video chunks, public APIs |
| Reverse Proxy Cache | Frequently requested origin responses | Public pages and expensive responses |
| Application Cache | Business objects | Users, Products, Metadata, Permissions |
| Distributed Cache | Shared application data across servers | Frequently accessed business data |

# Stale Data Across Multiple Cache Layers

Now another question naturally arises.

Suppose:

```text
logo.png
```

is cached in:

- Browser Cache
- CDN Cache
- Reverse Proxy Cache

Now YouTube changes the logo.

How do these caches know?

The answer is:

**They don't.**

Caches cannot magically know that the origin data has changed.

If nobody tells them anything,

they continue serving the old response.

This is exactly why stale data exists.

---

# TTL Helps...

One possible solution is TTL.

Suppose:

```
TTL = 1 Hour
```

After one hour,

the cache expires.

The next request fetches fresh data from the origin.

This works well.

But it introduces another problem.

Suppose YouTube changes its logo immediately.

Should users continue seeing the old logo for another hour?

Probably not.

TTL alone is not enough.

---

# Active Invalidation

From first principles, another solution naturally appears.

Instead of waiting for TTL,

the origin can notify caches:

> "Delete this object immediately."

Once deleted,

the next request fetches the latest version.

This idea is called **Cache Invalidation**.

We intentionally do **not** study it here.

Reason:

The moment we ask:

- How are thousands of CDN servers notified?
- What if one server is offline?
- What if only half the servers receive the notification?
- What if the notification is lost?

we enter the world of **Distributed Systems**.

Cache Invalidation deserves its own dedicated discussion in the next chapter.

For now,

remember only the intuition:

There are two broad ways to stop serving stale data.

1. Passive Expiration
   - Wait for TTL.
   - Simple.
   - Eventually fresh.

2. Active Invalidation
   - Origin explicitly removes stale cache.
   - Faster.
   - More complex.

# Final Mental Model

Everything we studied in this chapter can be summarized in one principle.

Every cache layer exists to prevent unnecessary work from reaching the next layer.

```text
Request

│
├── Browser Cache
│      ↓ Miss
├── CDN Cache
│      ↓ Miss
├── Reverse Proxy Cache
│      ↓ Miss
├── Application Cache
│      ↓ Miss
├── Distributed Cache
│      ↓ Miss
└── Database
```

The earlier a request is answered,

the cheaper it becomes.

This is the central idea behind modern caching architectures used by companies like:

- YouTube
- Netflix
- Amazon
- WhatsApp
- Discord

The layers don't compete.

They cooperate.

Each one asks exactly one question:

> **"Can I answer this request so that the next layer doesn't have to do any work?"**

If yes,

the request stops there.

Otherwise,

it continues deeper into the system.
