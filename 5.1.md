# Chapter 5.1 — Foundations of Caching

> **Caching is not about making applications faster. It is about preventing expensive work from happening repeatedly.**

---

# Why Was Caching Invented?

Most people think caching exists because databases are slow.

That's incorrect.

Modern databases are extremely fast.

The real problem is **repeating the same expensive work again and again**, even when the answer hasn't changed.

Consider Amazon's **Top 10 Bestselling Books**.

Millions of users request the exact same page.

Without caching, every request follows the same path.

```text
User
   │
   ▼
Application
   │
   ▼
Database
   │
   ▼
Application
   │
   ▼
User
```

Every request generates exactly the same response, yet the database performs the same work millions of times.

The problem isn't the query itself.

The problem is **repeated work**.

---

# What Actually Happens During A Database Read?

A database read is much more than simply fetching rows.

For every request, the database performs work such as:

- Accepting/using a connection.
- Authenticating the client.
- Authorizing permissions.
- Parsing the SQL query.
- Creating an execution plan.
- Reading indexes/pages.
- Fetching data.
- Preparing the response.
- Sending the response back.

Even if the same query is executed one million times, the database repeats almost all of this work one million times.

So the cost isn't just:

```text
Query Time
```

It is also:

- Connection overhead.
- Authentication overhead.
- CPU scheduling.
- Memory allocation.
- Network overhead.

During our discussion we realized something important.

Even before reading data, the database has already spent CPU cycles.

Repeated identical requests unnecessarily consume:

- CPU
- RAM
- Connection slots
- Network bandwidth
- Query execution resources

---

# Bigger Database Isn't The Solution

A natural solution is:

> "Let's buy a bigger database."

This certainly increases capacity.

More CPU.

More RAM.

Faster disks.

But notice what hasn't changed.

The database is still repeating identical work.

We've increased capacity.

We haven't eliminated waste.

Good engineering asks:

> **Should the database even be doing this work repeatedly?**

---

# The Core Idea Behind Caching

Imagine your friend asks:

> "Who won the Cricket World Cup?"

You answer.

Five seconds later another friend asks the same question.

Do you search Google again?

No.

You simply remember the answer.

Your brain naturally avoids repeating expensive work.

Caching follows exactly the same principle.

```text
Expensive Work

↓

Remember Result

↓

Reuse Result
```

A cache is simply temporary storage that remembers the result of expensive work so that identical work doesn't have to be repeated.

Notice the wording carefully.

A cache doesn't make the database faster.

A cache makes the application ask the database **less often**.

---

# Cache Hit

Suppose a request arrives.

Instead of directly contacting the database, the application first checks the cache.

```text
Application

↓

Cache

↓

Found

↓

Response
```

Since the answer already exists, the database is never contacted.

This is called a **Cache Hit**.

A cache hit saves:

- Database CPU.
- Database memory.
- Network traffic.
- Connection overhead.
- Query execution.

The first request paid the cost.

Every subsequent request reuses that work.

---

# Cache Miss

Now imagine another request.

The application checks the cache.

```text
Application

↓

Cache

↓

Not Found
```

The application now has no choice.

It asks the database.

```text
Application

↓

Cache

↓

MISS

↓

Database

↓

Application

↓

User
```

This is called a **Cache Miss**.

A cache miss is completely normal.

It simply means:

> "We don't currently remember this answer."

---

# What Should Happen After A Cache Miss?

Suppose the application simply returns the database response.

What happens when another user requests the same resource?

The database repeats the exact same work again.

Nothing has improved.

A better idea is obvious.

```text
Cache Miss

↓

Database

↓

Store Result In Cache

↓

Return Response
```

Now the first request performs two jobs:

1. Serve the current user.
2. Prepare the system for future users.

Every subsequent request becomes a cache hit.

This naturally derives one of the most common caching strategies:

> Fetch → Store → Reuse

---

# The Real Benefit Of Caching

Caching does **not** reduce the cost of producing the answer.

The first request still performs all the expensive work.

Caching reduces **how often** that expensive work happens.

This distinction is extremely important.

The database query isn't faster.

The database query simply happens less frequently.

---

# Where Should The Cache Live?

Suppose we have one application server.

The easiest solution is storing cached data directly inside the application's memory.

```text
           Application

               │

        Local Memory (RAM)

               │

        Cached Objects
```

This is called a **Local Cache**.

Reading from local RAM is extremely fast because no additional network communication is required.

---

# Scaling Changes Everything

Now imagine traffic increases.

One application server is no longer enough.

We launch four servers.

```text
               Users

                  │

      -------------------------

      │        │        │       │

     App1     App2     App3    App4
```

Suppose App1 fetches the bestseller list.

```text
App1

↓

Database

↓

Store In Local Cache
```

Now another request reaches App2.

Can App2 access App1's memory?

No.

Every application's RAM is private.

So App2 also contacts the database.

Soon every application server stores an identical copy.

```text
App1

Top 10 Books

------------

App2

Top 10 Books

------------

App3

Top 10 Books

------------

App4

Top 10 Books
```

The data is identical.

The memory is duplicated.

---

# Advantages Of Local Cache

## Extremely Fast

Everything happens within the same machine.

No network latency.

No additional service.

No serialization.

---

## Very Simple

No extra infrastructure.

No deployment.

No maintenance.

The application simply stores objects in RAM.

---

## Independent

If one application server crashes, the others continue functioning.

Each server owns its own cache.

---

# Problems With Local Cache

## 1. Duplicate Memory

If every server stores 2 GB of cached data,

```text
2 GB × 4 Servers = 8 GB
```

The same information is stored multiple times.

As the number of application servers grows, memory duplication also grows.

---

## 2. Cache Consistency

Suppose the database changes.

App1 updates its cache.

App2 doesn't.

App3 doesn't.

App4 doesn't.

Now users may receive different responses depending on which application server handles their request.

Caching solved one problem.

Repeated expensive work.

But introduced another.

Keeping multiple copies synchronized.

This pattern repeats throughout distributed systems.

Every optimization introduces new engineering challenges.

---

# Shared Cache

Instead of every application server maintaining its own cache, we can create one shared cache.

```text
                 Shared Cache

          +---------------------+
          |                     |
          | Cached Objects      |
          |                     |
          +---------------------+

             ▲    ▲    ▲    ▲

             │    │    │    │

           App1 App2 App3 App4
```

Every application server now reads from the same cache.

Benefits:

- Only one copy of the data.
- No duplicate memory.
- Every application server sees the same cached value.

This solves many of the problems introduced by local caching.

---

# Shared Cache Tradeoffs

A shared cache also introduces new costs.

Unlike local cache,

```text
Application

↓

Local RAM
```

the request now becomes

```text
Application

↓

Network

↓

Shared Cache

↓

Network

↓

Application
```

We've introduced:

- Network latency.
- Network failures.
- Connection management.
- Another service to maintain.
- Another component that can fail.

Again, we haven't eliminated tradeoffs.

We've simply chosen different ones.

---

# Local Cache vs Shared Cache

| Local Cache | Shared Cache |
|-------------|--------------|
| Fastest possible reads | Slightly slower (network hop) |
| Duplicate memory | Single copy of data |
| Every server independent | Centralized cache |
| Harder consistency | Easier consistency |
| No extra infrastructure | Extra infrastructure |
| Fewer network calls | Additional network communication |

There is no universally correct answer.

Choose based on:

- Application architecture.
- Memory requirements.
- Traffic patterns.
- Consistency requirements.
- Operational complexity.

The most important lesson from this section is:

> **Caching is not about storing data. Caching is about avoiding repeated expensive work.**

Everything else—cache hits, cache misses, local caches, shared caches—is simply an implementation of that single idea.

# Database As The Source Of Truth

Once we introduce caching, we now have **two copies** of the same data.

```text
Database

↓

Correct Data
```

```text
Cache

↓

Copy Of Database Data
```

The obvious question is:

> **Which copy should the application trust?**

The answer should always be:

> **Database = Source of Truth**
>
> **Cache = Performance Optimization**

The cache exists only to reduce expensive database reads.

It should never become the authoritative source of data.

A good mental model is:

```text
Database

↓

Owns Correctness

Cache

↓

Owns Performance
```

If the cache disappears, the application becomes slower.

If the database disappears, correctness is lost.

That is why the database must always remain the source of truth.

---

# Cache Consistency

Once multiple copies exist, another problem naturally appears.

```text
Database

↓

Version 2
```

```text
Cache

↓

Version 1
```

The cache is now **stale**.

Caching didn't create this problem.

Having **multiple copies** created this problem.

Any distributed system that stores the same information in multiple places eventually faces the same challenge.

---

# Updating Data

Suppose a seller updates a product price.

The application now needs to update:

```text
Database

+

Cache
```

The obvious question becomes:

> Which one should be updated first?

---

# Option 1 — Update Cache First

```text
Update Cache

↓

Update Database
```

Looks reasonable.

Now imagine the application crashes.

```text
Update Cache

✔ Success

↓

Application Crashes

↓

Database Never Updated
```

Result:

```text
Cache

↓

New Value
```

```text
Database

↓

Old Value
```

Users continue reading the new value from the cache even though the database never accepted the update.

Eventually the cache expires and users suddenly see the old value again.

The cache became **more correct than the database**, which should never happen.

---

# Option 2 — Update Database First

```text
Update Database

↓

Update Cache
```

Now imagine another crash.

```text
Database Updated

✔ Success

↓

Application Crashes

↓

Cache Not Updated
```

Result:

```text
Database

↓

New Value
```

```text
Cache

↓

Old Value
```

Some users may temporarily receive stale data.

However, the database still contains the correct information.

The cache can always be rebuilt.

Correctness survives.

Performance suffers.

Whenever correctness and performance conflict,

**correctness should usually win.**

This is why applications generally treat the database as the authoritative source.

---

# Stale Data

Stale data simply means:

> The database contains a newer value than the cache.

Example:

```text
Database

Price = ₹799
```

```text
Cache

Price = ₹899
```

The cache isn't corrupted.

It simply hasn't been updated yet.

This situation is unavoidable whenever multiple copies exist.

The goal is not to eliminate stale data.

The goal is to control it.

---

# Is Stale Data Always Bad?

No.

Different applications tolerate different levels of staleness.

Caching is fundamentally a business decision.

---

## Example 1 — Product Description

Yesterday:

```text
600 Pages
```

Today:

```text
620 Pages
```

A user still sees:

```text
600 Pages
```

Not ideal.

But usually acceptable.

---

## Example 2 — Profile Picture

A user changes their profile picture.

Other users continue seeing the previous image for a few minutes.

Again,

acceptable for most applications.

---

## Example 3 — Weather

Suppose weather information is cached.

If the temperature shown is thirty minutes old,

most users still receive useful information.

Performance is usually more valuable here.

---

## Example 4 — Stock Prices

Suppose the database contains:

```text
₹100
```

The cache still contains:

```text
₹97
```

Users may make financial decisions using outdated information.

Freshness becomes significantly more important.

---

## Example 5 — Banking

Suppose the actual account balance is:

```text
₹15,000
```

The cache still contains:

```text
₹40,000
```

Now stale data directly affects money.

Correctness becomes far more important than performance.

Applications like banking generally avoid serving stale values whenever possible.

---

# Freshness vs Performance

Every application chooses a position somewhere on this spectrum.

```text
Freshness <-------------------------------------------> Performance
```

Moving toward freshness means:

- More database reads.
- Higher infrastructure cost.
- Higher latency.
- Lower chance of stale data.

Moving toward performance means:

- More cache hits.
- Lower latency.
- Lower database load.
- Higher chance of stale data.

Neither side is universally correct.

Business requirements determine the tradeoff.

---

# Why TTL Exists

Suppose cached objects never expire.

```text
Day 1

↓

Product Cached
```

Six months later,

nobody has accessed that product.

Without expiration,

the cache still stores it.

Eventually thousands of unused entries occupy memory forever.

One reason TTL exists is obvious.

> Free unused memory.

But memory isn't the only reason.

---

# TTL Also Acts As A Recovery Mechanism

Consider another failure.

```text
Database Updated

✔ Success

↓

Application Crashes

↓

Cache Never Updated
```

Now:

```text
Database

↓

Latest Value
```

```text
Cache

↓

Old Value
```

Without TTL,

the stale value might remain forever.

With TTL,

the entry eventually expires.

The next request fetches fresh data from the database and rebuilds the cache.

TTL therefore provides **self-healing**.

It doesn't prevent stale data.

It limits **how long stale data can survive.**

---

# Choosing A TTL

There is no universal TTL.

The correct TTL depends on:

- How often the data changes.
- How expensive a cache miss is.
- How expensive stale data is.
- Business requirements.
- User expectations.

Ask yourself one question:

> **For how long can users safely observe outdated information?**

The answer usually determines the TTL.

---

# Example TTLs

These aren't rules.

They're engineering decisions.

| Data | Possible TTL | Reason |
|------|--------------|--------|
| Amazon Homepage | 12–24 Hours | Doesn't change frequently |
| Profile Picture | ~30 Days | Rarely updated |
| Weather | 30–60 Minutes | Doesn't change every second |
| Stock Prices | 1–2 Seconds | Freshness is critical |

Notice something important.

TTL is usually determined by **business requirements**, not technology.

---

# Engineering Conclusions

### Database

- Source of Truth.
- Owns correctness.

### Cache

- Performance optimization.
- Temporary copy of database data.

### Stale Data

- Unavoidable whenever multiple copies exist.
- Should be controlled, not expected to disappear completely.

### TTL

- Frees unused memory.
- Limits stale data lifetime.
- Acts as a self-healing mechanism after synchronization failures.

### Mental Model

```text
Database Updated

↓

Cache Eventually Updated

↓

Until then

↓

Application chooses one of two tradeoffs

↓

Serve Stale Data

OR

Wait For Fresh Data
```

The next question naturally becomes:

> **What happens when the cache expires and thousands of users request the same data simultaneously?**

# Cache Expiry & The Stampede Problem

TTL ensures that cached data eventually expires.

When a cache entry expires, the next request can no longer use it.

Instead, the application must rebuild the cache.

For a single request, the flow is straightforward.

```text
Application

↓

Cache

↓

MISS

↓

Database

↓

Store In Cache

↓

Response
```

Nothing seems wrong.

The interesting problems appear when traffic is high.

---

# What Happens After Cache Expiry?

Suppose Amazon caches its **Top 10 Bestselling Books** for one hour.

For one hour, every request is a cache hit.

```text
Application

↓

Cache

↓

HIT

↓

Response
```

The database isn't involved.

Exactly one hour later, the cache expires.

Now imagine **100,000 users** open the homepage almost simultaneously.

Every request follows the same flow.

```text
Request 1

↓

Cache Miss

↓

Database
```

```text
Request 2

↓

Cache Miss

↓

Database
```

```text
Request 3

↓

Cache Miss

↓

Database
```

...

```text
Request 100000

↓

Cache Miss

↓

Database
```

Every request asks for the **same data**.

Every request generates the **same SQL query**.

Every request receives the **same response**.

The database ends up repeating identical work thousands of times.

We've recreated the exact problem caching was supposed to solve.

---

# Why Is This Dangerous?

Suppose the database normally handles:

```text
20,000 Queries / Second
```

After one popular cache entry expires,

the database suddenly receives:

```text
120,000 Queries / Second
```

Traffic didn't increase.

Users didn't suddenly become six times more active.

The only thing that changed was:

```text
Cache Entry Expired
```

One expired object can temporarily overload the database.

This phenomenon is commonly known as a **Cache Stampede**.

A cache stampede occurs when many requests simultaneously miss the cache and independently rebuild the same data, creating a sudden spike in database load.

---

# The Core Observation

Ask yourself one question.

How many different answers does the database actually need to produce?

One.

Every request wants the same bestseller list.

There is absolutely no reason for the database to generate that answer thousands of times.

Repeated work is waste.

Caching exists to eliminate repeated work.

Therefore,

allowing every request to rebuild the cache defeats the entire purpose of caching.

---

# A Better Approach

Instead of allowing every request to query the database,

allow **only one request** to rebuild the cache.

```text
First Request

↓

Cache Miss

↓

Database

↓

Update Cache

↓

Response
```

Every other request simply notices that the cache is already being rebuilt.

```text
Request 2

↓

Cache

↓

Fetching...
```

```text
Request 3

↓

Cache

↓

Fetching...
```

```text
Request 4

↓

Cache

↓

Fetching...
```

Eventually,

the first request stores the fresh value.

After that,

every waiting request receives the same cached response.

The expensive database query executes exactly once.

This is precisely the intuition we discussed during the chapter.

---

# Thinking Of Cache As More Than Data

Initially we treated the cache as a place that stores values.

For example,

```text
Top10Books

↓

Book List
```

During a cache rebuild,

the cache may temporarily store something else.

```text
Top10Books

↓

FETCHING
```

Instead of saying:

> "The answer exists."

the cache now says:

> "Someone is already fetching the answer."

This tiny piece of information prevents thousands of duplicate database queries.

---

# Request Coordination

The cache now behaves like this.

```text
Application

↓

Check Cache
```

If:

```text
READY
```

Return cached value.

If:

```text
FETCHING
```

Don't contact the database again.

Someone else is already doing that work.

If:

```text
NOT FOUND
```

Become the request responsible for rebuilding the cache.

This idea is much more important than the implementation itself.

Whether the system uses locks, promises, futures, request coalescing, or another coordination mechanism is an implementation detail.

The engineering principle remains the same.

> **Never perform expensive work twice if another request is already doing it.**

---

# What Should Other Requests Do?

Suppose Request 1 is rebuilding the cache.

What should Requests 2–100,000 do?

There are two common strategies.

---

## Option 1 — Wait

```text
Cache

↓

FETCHING

↓

Wait

↓

Cache Updated

↓

Fresh Response
```

Advantages:

- Latest data.
- No stale information.
- Database queried only once.

Disadvantages:

- Higher response latency.
- Users wait while the cache is rebuilt.

---

## Option 2 — Serve Stale Data

Instead of waiting,

the application temporarily returns the previous cached value.

```text
Old Cached Value

↓

Immediate Response
```

Advantages:

- Very low latency.
- Excellent user experience.
- Database queried only once.

Disadvantages:

- Users may temporarily receive outdated information.

---

# Which Strategy Is Better?

There is no universal answer.

The correct strategy depends on the application.

---

## Banking

Users expect the latest account balance.

Waiting a few milliseconds is usually acceptable if it guarantees correctness.

Correctness is more valuable than latency.

---

## Payments

Fresh information is generally preferred.

Serving stale transactional data can produce incorrect business behaviour.

---

## Amazon Homepage

Showing yesterday's bestseller list for another few hundred milliseconds is usually acceptable.

Most users won't even notice.

Performance is generally more valuable than perfect freshness.

---

## Weather

A slightly older forecast is often perfectly usable.

Fast responses are usually preferred over absolute freshness.

---

# Mental Model

Notice how our understanding of caching has evolved.

Initially we thought:

```text
Caching

↓

Store Data
```

Now we realize caching is actually about:

```text
Avoid Repeated Expensive Work
```

Sometimes that means:

- Storing data.

Sometimes it means:

- Coordinating requests.

Sometimes it means:

- Serving stale data.

The implementation changes.

The principle never does.

---

# Engineering Conclusions

### Cache Stampede

Occurs when many requests simultaneously miss the cache and independently rebuild identical data.

Result:

- Sudden database spike.
- Repeated expensive work.
- Temporary loss of caching benefits.

---

### Request Coordination

Only one request should rebuild the cache.

Other requests should recognize that rebuilding is already in progress instead of repeating the same work.

---

### FETCHING State

The cache doesn't always store data.

Sometimes it stores the state of that data.

Example:

```text
READY

FETCHING

NOT FOUND
```

This allows the application to coordinate requests efficiently.

---

### Wait vs Serve Stale

| Wait | Serve Stale |
|------|-------------|
| Latest data | Lower latency |
| Higher response time | Slightly stale responses |
| Preferred for correctness | Preferred for user experience |

Neither strategy is universally correct.

Choose according to business requirements.

# Cache Eviction, Mental Models & Engineering Principles

So far, we've answered:

- Why caching exists.
- Why repeated work is expensive.
- Cache Hits and Cache Misses.
- Local vs Shared Cache.
- Database as the Source of Truth.
- Stale Data.
- TTL.
- Cache Stampede.
- Request Coordination.

One final question remains.

> **What happens when the cache becomes full?**

Memory is not infinite.

No cache can store everything forever.

Eventually the cache must decide:

> **What should stay?**
>
> **What should be removed?**

This process is called **Cache Eviction**.

---

# Why Cache Eviction Exists

Imagine a cache with space for only three objects.

```text
Cache

────────────

Product A

Product B

Product C

────────────
```

Now another request arrives.

```text
Product D
```

Where should it go?

The cache is already full.

Something has to leave.

Caching is no longer only about storing data.

It's also about deciding what deserves memory.

---

# Which Data Should Stay?

There is no universally correct answer.

Instead, engineers look at **access patterns**.

During our discussion we arrived at two important intuitions.

---

## 1. Recency

Suppose a user accesses:

```text
Product A
```

five seconds ago.

Another product was last accessed:

```text
Three Months Ago
```

If memory becomes full,

which one is more likely to be useful again?

Usually,

the recently accessed one.

This leads to the intuition of keeping **recently used** data.

---

## 2. Frequency

Now imagine another situation.

One product has been requested:

```text
1,000,000 Times
```

Another has been requested:

```text
Once
```

Even if both were accessed recently,

the highly popular object is generally more valuable to keep.

This leads to another intuition.

Keep data that is requested **frequently**.

---

# There Is No Universal Policy

Some applications benefit more from recency.

Others benefit more from frequency.

The correct policy depends on how users access data.

During our discussion we concluded:

> Observe access patterns.

If users repeatedly access the same objects,

keeping those objects cached provides the greatest benefit.

Caching is driven by workload,

not by fixed rules.

---

# Complete Request Lifecycle

Putting everything together, a typical read request now looks like this.

```text
                 Client
                    │
                    ▼
             Application
                    │
                    ▼
             Check Cache
                    │
     ┌──────────────┼──────────────┐
     │              │              │
     ▼              ▼              ▼
   HIT         FETCHING          MISS
     │              │              │
     │        Wait / Serve         │
     │         Stale Data          │
     │              │              ▼
     │              │         Database
     │              │              │
     │              │              ▼
     │              └────── Store In Cache
     │                             │
     └─────────────────────────────▼
                             Return Response
```

Every optimization we've discussed fits somewhere in this flow.

---

# Read Path Summary

A read request should generally follow this order.

```text
1. Check Cache

↓

2. Cache Hit?

↓

Yes

↓

Return Cached Data

OR

↓

No

↓

Read Database

↓

Store Result In Cache

↓

Return Response
```

The database is contacted **only when necessary**.

Everything else attempts to reuse previous work.

---

# Write Path Summary

For write operations, the database always remains authoritative.

```text
Write Request

↓

Update Database

↓

Update / Invalidate Cache

↓

Return Response
```

Updating the database first ensures that correctness is never sacrificed for performance.

If the cache update fails,

it can always be rebuilt from the database.

The reverse is not true.

## 4. Business Requirements Drive Caching Decisions

Caching decisions are rarely purely technical.

Questions such as:

- Can users see stale data?
- How fresh must the data be?
- Is lower latency more valuable than absolute correctness?
- How long should cached data live?

are fundamentally business questions.

Technology simply implements those decisions.

---

## 5. Caching Is About Tradeoffs

Every major decision introduces tradeoffs.

| Decision | Tradeoff |
|----------|----------|
| Local Cache | Faster reads vs duplicate memory |
| Shared Cache | Single copy vs network overhead |
| Long TTL | Better performance vs more stale data |
| Short TTL | Fresher data vs more database load |
| Wait | Fresh data vs higher latency |
| Serve Stale | Lower latency vs outdated information |

There is no universally correct answer.

Choose according to the application's requirements.

---

# Common Misconceptions

### ❌ "Caching makes database queries faster."

✔ No.

Caching reduces how often database queries execute.

---

### ❌ "The cache should always contain the latest data."

✔ Not necessarily.

Many systems intentionally serve slightly stale data because the performance benefits outweigh the temporary inconsistency.

---

### ❌ "TTL exists only to free memory."

✔ TTL also limits the lifetime of stale data and provides a recovery mechanism if cache synchronization fails.

---

### ❌ "A Cache Miss Means Something Is Wrong."

✔ Cache misses are completely normal.

The first request after startup, expiration, or eviction will naturally miss the cache.

---

### ❌ "Shared Cache Is Always Better."

✔ Shared caches reduce duplicate memory but introduce additional network communication, infrastructure, and operational complexity.

# Engineering Principles To Remember

- Don't repeat expensive work.
- Cache only when repeated work exists.
- The database is always the source of truth.
- Correctness is generally more important than performance.
- Stale data is inevitable when multiple copies exist.
- TTL limits stale data; it doesn't eliminate it.
- Coordinate requests to prevent cache stampedes.
- Choose caching strategies based on business requirements.
- Every optimization introduces new tradeoffs.
- Design for scalability, not just today's traffic.
