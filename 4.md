# HTTP STATUS CODES

## Why Do Status Codes Exist?

Status codes are not for humans.

They are infrastructure signals.

Used by:

- Browsers
- Mobile Apps
- API Gateways
- Load Balancers
- Retry Systems
- Monitoring Systems

Without status codes every system would need to parse custom text.

---

## Categories

### 1xx

Informational.

Rarely used.

### 2xx

Success.

Examples:

- 200 OK
- 201 Created
- 204 No Content

### 3xx

Redirection.

Examples:

- 301
- 302

### 4xx

Client error.

Client must fix something.

### 5xx

Server error.

Server must fix something.

---

## Why Not Return 200 For Everything?

Bad:

200 OK

{
  "success": false
}

Why bad?

Monitoring sees success.

Retries become difficult.

Observability breaks.

Status codes communicate outcome to machines.

---

## 401 vs 403

401

"I don't know who you are."

Examples:

- Missing JWT
- Invalid JWT
- Expired Session

403

"I know who you are.
You are not allowed."

Example:

Normal user calling admin endpoint.

---

## Why Some Systems Return 404 Instead Of 403

Security.

Returning 403 confirms resource exists.

Returning 404 hides existence.

---

## Retry Thinking

Usually Retry:

- 500
- 502
- 503

Usually Don't Retry:

- 400
- 401
- 403
- 404

---

## Real Example

Payment API.

Client sends request.

Payment succeeds.

Response lost.

Client retries.

Now status codes alone aren't enough.

This is where:

- Idempotency
- State reconciliation

become important.

# COOKIES

## What Is A Cookie?

A small piece of data stored by browser.

Automatically attached to matching requests.

Example:

Set-Cookie:

sessionId=abc123

Future requests:

Cookie: sessionId=abc123

---

## Biggest Misconception

Cookie != Authentication

Cookie != Session

Cookie != Login

Cookie is merely a transport mechanism.

Think:

Cookie = Envelope

Contents may be:

- Session ID
- JWT
- User Preferences
- Shopping Cart

---

## Why Cookies Were Invented

HTTP is stateless.

Request 1:

GET /products

Request 2:

GET /cart

Server doesn't naturally know they're same user.

Cookies introduce continuity.

---

## Why Cookies Are Dangerous

Browser automatically sends them.

Convenient.

But creates risks:

- CSRF
- Session Hijacking

---

## HttpOnly

JavaScript cannot read cookie.

Protects against token theft via XSS.

---

## Secure

Cookie sent only via HTTPS.

---

## SameSite

Helps prevent CSRF.

---

## Real Engineering Question

Why store JWT in HttpOnly Cookie instead of LocalStorage?

Because XSS can read LocalStorage.

HttpOnly blocks JavaScript access.

---

# SESSIONS

## What Is A Session?

Server-side authentication state.

Browser stores:

sessionId

Server stores:

- User
- Permissions
- Login State

---

## Hotel Analogy

Session ID = Room Card

Server = Hotel Database

Card identifies you.

Actual information lives elsewhere.

---

## Flow

Login

↓

Server creates session

↓

Stores session

↓

Returns session ID

↓

Browser sends session ID

↓

Server performs lookup

---

## Why Sessions Feel Natural

Server remains in control.

Logout easy.

Revocation easy.

Permission updates easy.

---

## Scaling Problem

Single Server:

Easy.

Multiple Servers:

Where is session?

Request 1:

Server A

Request 2:

Server B

Problem.

---

## Solutions

### Sticky Sessions

Same user always reaches same server.

Problem:

Server dies.

Session effectively dies.

### Shared Session Store

Usually Redis.

Most common production solution.

---

## Failure Modes

Redis Down.

Users logged out.

Authentication degraded.

---

## Why Large Companies Still Use Sessions

Easy control.

Easy forced logout.

Easy security operations.

---

# JWT

## Why JWT Was Invented

Sessions require central storage.

Distributed systems dislike central dependencies.

JWT removes lookup requirement.

---

## Mental Model

Session:

Server remembers.

JWT:

User carries proof.

---

## Structure

Header

Payload

Signature

---

## Important Truth

JWT is usually signed.

Not encrypted.

Anyone can read payload.

They cannot modify it.

---

## Why JWT Scales Better

Microservices.

User Service

Payment Service

Order Service

Inventory Service

All can validate token independently.

No session lookup.

---

## Why JWT Revocation Is Hard

User logs out.

Token still exists.

Server cannot magically delete every copy.

Common solution:

- Short-lived access token
- Refresh token

---

## Common Interview Trap

Is JWT better than Sessions?

Wrong question.

Correct question:

What problem are we solving?

Sessions optimize control.

JWT optimizes scalability.

---

# COOKIES VS SESSIONS VS JWT

Cookie

Transport.

Session

Server-side state.

JWT

Client-side state.

---

## Comparison

Sessions

Advantages:

- Easy revocation
- Easy logout
- Easy permission updates

Disadvantages:

- Shared storage
- Session lookup

JWT

Advantages:

- Stateless
- Scales well

Disadvantages:

- Revocation difficult
- Token leakage dangerous

---

# REST

## Why REST Became Popular

Before REST many APIs were custom.

REST standardized communication.

---

## Core Philosophy

Model resources.

Not actions.

Resources:

- Users
- Orders
- Products

Examples:

GET /users/123

POST /orders

DELETE /users/123

---

## Why Resource Thinking Matters

Resources exist regardless of operations.

User exists.

Order exists.

Product exists.

Operations happen to resources.

---

## REST Constraints

### Statelessness

Each request contains everything needed.

Benefit:

Any server can handle request.

### Uniform Interface

Consistency.

Developers know how APIs behave.

### Cacheability

Allows performance optimization.

### Client-Server Separation

Independent evolution.

---

## Why Statelessness Helps Scaling

Load balancer can send request anywhere.

No dependency on local memory.

Infrastructure simpler.

---

## Common Criticism

Some operations feel awkward.

Example:

approveLoan()

generateInvoice()

trainModel()

REST can become unnatural.

---

# RPC

## What Is RPC?

Remote Procedure Call.

Calling function over network.

Examples:

createUser()

approveLoan()

generateInvoice()

---

## REST vs RPC

REST

Resource-oriented.

RPC

Action-oriented.

---

## Why Companies Use gRPC

gRPC is modern RPC.

Advantages:

- Strong contracts
- Code generation
- Binary protocol
- Faster communication

---

## Public APIs vs Internal APIs

Public:

Usually REST.

Humans consume.

Internal:

Often gRPC.

Machines consume.

---

## Interview Question

Why don't companies expose gRPC publicly everywhere?

REST is easier:

- Debugging
- Tooling
- Browser support

---

# POLLING

## Problem

Server receives new data.

Client doesn't know.

HTTP cannot push updates.

---

## Solution

Ask repeatedly.

Every few seconds:

GET /notifications

---

## Tradeoff

Small interval:

+ Better latency
- More load

Large interval:

+ Less load
- Worse latency

---

## Why Polling Doesn't Scale Well

Imagine:

1 million users

Polling every second

Massive waste.

Most responses:

No updates.

---

# LONG POLLING

## Key Insight

Ask once.

Wait.

---

Client Request

↓

Server waits

↓

Event happens

↓

Server responds

---

## Why Invented?

Reduce useless polling requests.

---

## Problems

- Open connections
- Timeouts
- Memory usage
- Reconnection complexity

---

## Historical Context

Long Polling was a bridge.

Better than Polling.

Before WebSockets became mainstream.

---

# WEBSOCKETS

## Why Invented?

Polling wasteful.

Long Polling awkward.

Need persistent communication.

---

## Mental Model

HTTP

Question → Answer

WebSocket

Conversation

---

## Flow

HTTP Request

↓

Upgrade Connection

↓

Persistent TCP Connection

↓

Bidirectional Communication

---

## Real Examples

- WhatsApp
- Slack
- Discord
- Google Docs
- Multiplayer Games
- Stock Tickers

---

## Why WhatsApp Doesn't Use Polling

Imagine:

500M users

Polling every second.

Infrastructure disaster.

WebSockets allow server push.

---

## Deep Scaling Problem

User connected to Server A.

Notification generated on Server B.

How does B reach user?

Solutions:

- Redis Pub/Sub
- Kafka
- Message Brokers

This is where distributed systems begin.

---

## Why WebSockets Are Harder To Scale

REST:

Any request can hit any server.

WebSockets:

Connection stays attached to specific server.

Now connections become state.

Infrastructure complexity increases.

---

## Failure Modes

- Connection drops
- Mobile network changes
- Server restart
- Reconnection storms
- Load balancer issues

---

# POLLING VS LONG POLLING VS WEBSOCKETS

Polling

+ Simple
- Wasteful

Long Polling

+ Better latency
+ Fewer requests

- Open connection overhead

WebSockets

+ True real-time
+ Efficient

- Stateful
- Operational complexity

---

# ENGINEERING TAKEAWAYS

Status Codes:
Infrastructure signals.

Cookies:
Transport mechanism.

Sessions:
Server remembers.

JWT:
Client carries proof.

REST:
Resource-oriented design.

RPC:
Action-oriented design.

Polling:
Repeated asking.

Wait for answer.

WebSockets:
Persistent conversation.

Most important lesson:
