# HTTP STATUS CODES

## The Real Reason Status Codes Exist

Many beginners think status codes are for humans.

Wrong.

They are primarily for machines.

Imagine:

Client

â†“

â†“

â†“
Monitoring System
â†“

All these components need a standardized way to understand outcomes.

A status code is a machine-readable summary of what happened.

---

## Why Not Always Return 200?

Bad:

200 OK

{
  "success": false
}

Why is this bad?

Monitoring systems now think everything is healthy.

Load balancers think requests succeeded.

Retry systems don't know what happened.

Observability breaks.

Status codes are infrastructure signals.

---

## 401 vs 403

This is asked frequently.

401:

"I don't know who you are."

Examples:

- Missing JWT
- Expired JWT
- Invalid Session

403:

"I know who you are.
You are not allowed."

Example:

Normal user trying:

DELETE /admin/users

---

## Why 404 Instead Of 403 Sometimes?

Security.

Suppose:

GET /admin/users

If server responds:

403

Attacker learns:

Admin endpoint exists.

Some systems intentionally return:

404

to hide existence.

---

## Why Are 5xx Dangerous?

A 500 usually means:

- Bug
- Database issue
- Dependency failure
- Infrastructure issue

A large spike in 5xx is often how production incidents are detected.

# COOKIES

## Most Important Misconception

Cookie != Login

Cookie != Session

Cookie != Authentication

Cookie is merely a transport mechanism.

Think:

Cookie = Envelope

What you place inside is your choice.

Examples:

- Session ID
- JWT
- Theme Preference
- Shopping Cart Data

---

## Why Were Cookies Invented?

HTTP is stateless.

Request #1:

GET /products

Request #2:

GET /cart

Server cannot naturally connect them.

Cookies introduced continuity.

---

## Why Does Browser Automatically Send Cookies?

Convenience.

Without cookies:

Every request would require manually attaching state.

Browser does it automatically.

This is both:

Advantage

and

Security risk.

---

## Why HttpOnly Matters

Without HttpOnly:

JavaScript can read cookie.

XSS Attack:

Attacker injects script.

Script steals token.

User account compromised.

HttpOnly prevents browser JavaScript from reading cookie.

---

## Real Production Question

If JWT can be stored in LocalStorage,

why do many companies still use cookies?

Answer:

HttpOnly cookies provide protection against token theft through XSS.

# JWT

## Why JWT Was Created

Sessions become painful in distributed systems.

Every service needs access to session store.

JWT removes that dependency.

---

## Real Mental Model

Session:

Server remembers user.

JWT:

User carries proof.

---

## Structure

Header
Payload
Signature

---

## Important Truth

JWT Payload Is Usually Visible

Example:

{
 "userId": 42,
 "role": "admin"
}

Anyone can decode it.

Signature prevents modification.

Not reading.

---

## Why JWT Scales Better

Microservices:

User Service
Payment Service
Order Service

Every service can verify JWT.

No central lookup required.

---

## Why Revocation Is Hard

User logs out.

Server cannot magically delete JWT from millions of devices.

Token still exists.

Common solution:

Short-lived access tokens.

---

## Why JWT Is Not Automatically Better

Interview trap.

JWT solves:

Scalability.

Sessions solve:

Control.

# REST

## Why REST Became Popular

Before REST:

Every company invented custom APIs.

Chaos.

REST standardized communication.

---

## Core Philosophy

Resources.

Not actions.

Bad:

createUser()

Good:

POST /users

Bad:

getProduct()

Good:

GET /products/123

---

## Why Resources Matter

Users

Orders

Products

exist regardless of actions.

REST models real-world entities.

---

## Why Statelessness Matters

Any request can hit any server.

Load balancing becomes easy.

Horizontal scaling becomes easy.

---

## Deep Question

Why is statelessness so valuable?

Because servers stop depending on local memory.

Infrastructure becomes simpler.

---

## REST Is Not About HTTP Methods

Common misconception.

REST is about:

Resources

Uniform interface

Statelessness

Cacheability

HTTP methods are merely implementation tools.

# POLLING

## The Problem

Server gets new information.

How does client know?

HTTP cannot push.

---

## Solution

Example:

---

## Real Problem

Most requests:

"No new notifications."

Resources wasted.

---

## Engineering Tradeoff

Smaller interval:

Better latency.

More server load.

Larger interval:

Less load.

Worse latency.
